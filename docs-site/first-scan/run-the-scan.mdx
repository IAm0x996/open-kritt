A **scan** ties everything together: it runs one workflow against a target repository with
the model, harness, post-scripts, and rankers you choose. The engine queues it and runs
every step.

<Frame caption="Creating a new scan: pick a workflow, choose a target, and configure the run.">
  <img src="/images/create-scan.png" alt="open·kritt new scan form" />
</Frame>

## Create the scan

Open **Scans → New scan** and work down the form:

<Steps>
  <Step title="Choose a workflow">
    Pick the workflow you built or one of the
    [built-in workflows](/workflows/built-in-workflows). The scan will run its steps in
    depth order.
  </Step>
  <Step title="Choose a target">
    Toggle **Remote** or **Local**. For remote, enter a GitHub `owner/repo` and a
    `commit_sha` (defaults to `HEAD`). For local, pick a folder you've mounted. Set
    `repo_scope` to describe what to focus on (defaults to "full repository").
  </Step>
  <Step title="Add dependencies and configuration (optional)">
    Add dependency repos to be checked out alongside the target, and a JSON
    **configuration** blob the agents can read. To give every task a cumulative second
    attempt, use `{"repeat_runs": 2}`. The second execution of each concrete task sees
    only that task's earlier output and returns new records to append; leave it at `1` for
    the cheapest first scan.
  </Step>
  <Step title="Pick model, provider, and harness">
    Choose a model from the searchable catalog for the selected provider. OpenRouter loads
    choices from its authenticated model catalog and also accepts an exact custom
    `author/model` ID when needed. Then choose a compatible harness and thinking effort.
    The form defaults Codex to the Codex harness and Anthropic/OpenRouter to Claude Code. If
    no provider is listed, complete
    [AI provider setup](/ai-provider-setup/overview) and recreate the stack.
    Keep **One model for all depths** for the normal path. Choose **Customize by depth**
    when, for example, discovery should use a faster model and deeper investigation
    should use a stronger one. Post-processing continues to use the default row.
  </Step>
  <Step title="Attach skills, post-scripts, and rankers">
    Select any agent skills, one or more post-scripts (the first is the primary), and one
    or more severity rankers plus optional scan-specific rules.
  </Step>
  <Step title="Launch">
    Submit. If another scan is active, choose **Start immediately** to run concurrently
    or **Queue** to wait until active scans finish.
  </Step>
</Steps>

The normal path is `pending`, optional `prewarming_cache`, `running`, `post_processing`,
then `completed`. A queued scan starts when the engine is idle. A temporarily
`rate_limited` scan retries automatically without discarding completed work. See
[errors and re-runs](/scan-results/errors-and-re-runs).

<Warning>
  Each workflow branch, repeat, ranking batch, and post-script consumes model calls.
  Start with one repeat and a narrow `repo_scope`, then expand after the workflow produces
  useful results.
</Warning>

## Watch it run

The scan detail page shows live status, active jobs, and any errors. As findings land,
they appear in the list - ranked by your ranker, enriched by your post-scripts.

## View the results

When the scan finishes, open any finding to see its full write-up, proof of concept, and
post-script output. See [Scan results](/scan-results/how-to-view) for the tour.
